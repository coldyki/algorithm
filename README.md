사용자 행동 및 키워드 기반 개인화 여행 코스 추천 서비스

## 2. 프로토타입 설명

GraphTrip은 사용자의 장소 저장/좋아요 데이터를 기반으로 여행 장소와 여행 코스를 추천해주는 서비스 프로토타입이다.

초기 서비스 단계에서는 특정 장소를 좋아한 사용자들이 함께 좋아요/저장한 다른 장소를 추천한다.

이후 사용자 데이터가 쌓이면 리뷰 작성 시 선택한 키워드, 예를 들어 감성적인, 조용한, 사진맛집, 가성비 등을 활용하여 사용자 취향에 맞는 장소와 여행 코스를 개인화 추천하는 방식으로 확장할 수 있다.

또한 추천된 장소들을 이동 거리 기준으로 재정렬하여 하나의 효율적인 여행 코스로 제공한다.

### What

사용자 취향에 맞는 여행 장소를 추천하고, 추천된 장소들을 이동 거리 기준으로 효율적인 여행 코스로 구성하는 서비스이다.

### Who

- 여행 계획을 쉽게 세우고 싶은 사용자
- 자신의 취향에 맞는 장소를 추천받고 싶은 사용자
- 인기순이 아닌 개인 맞춤형 여행 코스를 원하는 사용자
- 처음 방문하는 지역에서 효율적인 동선을 추천받고 싶은 사용자

### When

- 여행을 계획할 때
- 주말 나들이 코스를 찾을 때
- 특정 장소 방문 후 비슷한 분위기의 장소를 찾고 싶을 때
- 여행 일정과 이동 동선을 효율적으로 구성하고 싶을 때

### Why

기존 여행 추천 서비스는 단순 인기순 추천이나 지역 기반 추천이 많아 개인 취향을 충분히 반영하지 못하는 한계가 있다.

GraphTrip은 사용자 행동 데이터, 장소 간 연관성, 리뷰 키워드, 이동 거리 정보를 함께 활용하여 맞춤형 여행 추천을 제공하는 것을 목표로 한다.

## 4. Input 데이터

### 4.1 장소 데이터

파일 경로:

```text
data/places.json
```

데이터 내용:

- 경주 주요 여행지 ID
- 장소명
- 위도(lat)
- 경도(lng)
- 장소별 키워드

데이터 예시:

```json
{
  "id": 1,
  "name": "황리단길",
  "lat": 35.8362,
  "lng": 129.2181,
  "keywords": ["감성적인", "사진맛집", "카페투어", "데이트"]
}
```

출처:

팀에서 경주 주요 여행지를 기준으로 직접 구성한 프로토타입용 장소/좌표/키워드 데이터이다.

### 4.2 좌표 기반 거리 계산

본 모듈은 별도의 `distances.csv` 파일을 사용하지 않고, `places.json`의 `lat`, `lng` 좌표를 이용해 장소 간 직선거리를 계산한다.

계산된 직선거리는 Graph의 간선 가중치로 사용된다.

주의할 점:

본 프로토타입의 거리는 지도 API를 사용한 실제 도로 이동거리가 아니라 위도/경도 기반 근사 거리이다. 실제 서비스에서는 지도 API를 연결하여 실제 이동 거리와 이동 시간을 반영할 수 있다.

## 5. 사용한 자료구조와 알고리즘

### 5.1 사용 자료구조

Graph:

경주 여행지를 정점으로 표현하고, 장소 간 이동 거리를 간선의 가중치로 표현하기 위해 사용하였다. 여행 장소 간 연결 관계와 이동 거리를 함께 나타내는 데 적합하다.

Priority Queue:

Dijkstra 알고리즘에서 현재까지의 거리가 가장 짧은 장소를 빠르게 선택하기 위해 사용하였다. Python 표준 라이브러리의 `heapq`를 활용하였다.

Dictionary:

장소 ID를 기준으로 장소명, 키워드, 거리 정보, 최단거리 테이블을 빠르게 조회하기 위해 사용하였다.

Set:

Greedy 알고리즘에서 아직 방문하지 않은 추천 장소를 관리하기 위해 사용하였다. 방문 여부 확인과 중복 제거를 빠르게 처리할 수 있다.

List:

추천 장소 ID 목록과 최종 여행 코스처럼 순서가 중요한 데이터를 저장하기 위해 사용하였다.

### 5.2 사용 알고리즘

Dijkstra Algorithm:

현재 위치에서 다른 장소까지의 최단 거리를 계산하기 위해 사용하였다. 장소 간 이동 거리는 모두 양수이므로 Dijkstra 알고리즘을 적용하기 적합하다.

Greedy Algorithm:

현재 위치에서 아직 방문하지 않은 추천 장소 중 가장 가까운 장소를 다음 방문지로 선택하기 위해 사용하였다. 모든 방문 순서를 완전탐색하지 않고 빠르게 효율적인 코스를 생성할 수 있다.

Floyd-Warshall Algorithm:

모든 장소 쌍 사이의 최단 거리를 미리 계산하는 비교용 추가 기능으로 사용하였다. 본 프로토타입은 경주 주요 여행지 10개를 대상으로 하므로 전체 최단거리 테이블을 계산해도 부담이 작다.

Frequency Counting:

최종 코스에 포함된 장소들의 키워드 빈도를 계산하기 위해 보조적으로 사용하였다. 이를 통해 코스의 주요 키워드와 코스 유형을 출력할 수 있다.

주의할 점:

Greedy 알고리즘은 현재 위치에서 가장 가까운 장소를 선택하는 휴리스틱 방식이므로 항상 전역 최적해를 보장하지는 않는다. 하지만 빠르게 효율적인 코스를 생성할 수 있어 본 프로토타입의 목적에 적합하다.

## 6. 설계 및 구현 방법

### 6.1 팀원4 모듈의 역할

팀원4 모듈은 팀원1, 팀원2, 팀원3의 추천 시스템을 통해 선정된 장소 ID 목록을 입력받아, 이동 거리 기준으로 효율적인 여행 코스를 생성하는 후처리 모듈이다.

입력:

```text
start_place_id
recommended_place_ids
```

처리:

- 장소 데이터 로드
- places.json의 위도/경도 좌표로 장소 간 직선거리 계산
- 좌표 기반 무방향 가중치 그래프 생성
- Dijkstra 알고리즘으로 최단거리 계산
- Greedy 알고리즘으로 다음 방문 장소 선택
- 기존 추천 순서와 최적화된 코스의 이동 거리 비교
- 구간별 이동 거리와 개선율 계산
- 코스 키워드 분석

출력:

- 최적화된 여행 코스
- 총 이동 거리
- 구간별 이동 거리
- 기존 추천 순서 대비 거리 차이
- 개선율
- 코스 주요 키워드
- 코스 유형

### 6.2 전체 서비스 연결 흐름

```text
사용자 좋아요/저장 데이터
-> 키워드 기반 장소 분류
-> 개인화 추천 점수 계산
-> 추천 장소 ID 리스트 생성
-> 팀원4 여행 코스 최적화
-> 최종 여행 코스 출력
```

### 6.3 핵심 동작 방식

1. `places.json`에서 장소 ID, 장소명, 키워드, 위도, 경도를 읽는다.
2. 위도/경도 좌표를 이용해 장소 간 직선거리를 계산한다.
3. 계산된 거리를 간선 가중치로 사용하여 Graph를 만든다.
4. 추천 장소 ID 목록에서 중복을 제거한다.
5. 현재 위치에서 미방문 추천 장소까지의 최단 거리를 Dijkstra 알고리즘으로 계산한다.
6. Greedy 알고리즘으로 가장 가까운 미방문 장소를 다음 방문지로 선택한다.
7. 모든 추천 장소를 방문할 때까지 5~6번 과정을 반복한다.
8. 기존 추천 순서와 최적화된 코스의 총 이동 거리를 비교한다.

## 9. Lessons Learned

### Plus

여행 장소와 장소 간 이동 거리를 Graph로 표현하니 실제 여행 동선을 자연스럽게 모델링할 수 있었다.

Dijkstra 알고리즘을 통해 직접 연결되지 않은 장소 사이도 그래프상 최단거리로 계산할 수 있었고, Greedy 전략을 함께 사용하여 추천 장소들을 빠르게 효율적인 순서로 정렬할 수 있었다.

### Minus

현재 거리는 places.json의 위도/경도를 이용한 직선거리이기 때문에 실제 교통 상황, 도보 시간, 대중교통 시간 등을 완전히 반영하지 못한다.

또한 Greedy 방식은 매 순간 가장 가까운 장소를 선택하므로 빠르다는 장점은 있지만, 전체 이동 거리가 항상 가장 짧은 전역 최적해를 보장하지는 않는다.

### Interesting

추천 시스템이 좋은 장소를 골라주는 것과, 그 장소들을 실제로 방문하기 좋은 순서로 정렬하는 것은 서로 다른 문제라는 점이 흥미로웠다.

개인화 추천 결과에 경로 최적화 알고리즘을 결합하면 단순 장소 추천보다 실제 사용자가 바로 활용할 수 있는 여행 코스에 가까워진다는 점을 확인할 수 있었다.

## 10. 향후 확장 방향

- 실제 지도 API를 연결하여 장소 간 실제 이동 거리와 예상 이동 시간을 반영할 수 있다.
- 사용자의 여행 가능 시간, 선호 카테고리, 이동 수단을 입력받아 더 개인화된 코스를 생성할 수 있다.
- Greedy 방식의 한계를 보완하기 위해 TSP 기반 알고리즘을 적용할 수 있다.
- 최단거리 계산 결과에 중간 경유지를 포함하여 더 자세한 이동 경로를 출력할 수 있다.
- Streamlit 또는 Flask를 이용해 추천 장소 입력과 코스 결과를 시각적으로 확인할 수 있는 UI로 확장할 수 있다.
